# Lecture 2 — What the JVM Actually Does: Bytecode, the Heap/Stack, and Garbage Collection

> **Duration:** ~2 hours of reading + hands-on.
> **Outcome:** You can trace a `.java` file through `javac` to bytecode to JIT-compiled native code, read a `javap` disassembly at a high level, explain where objects and locals live (heap vs stack), and describe what a generational garbage collector does — well enough to debug an `OutOfMemoryError` or a slow endpoint in a later week.

If you only remember one thing from this lecture, remember this:

> **Java does not compile to machine code. It compiles to bytecode, which the JVM runs — interpreting it at first, then compiling the hot paths to native code at run time. Your objects live on a garbage-collected heap; your method calls and local variables live on per-thread stacks.** Once you hold that model, "why is this slow," "why did it OOM," and "why does my benchmark warm up" stop being mysteries.

You wrote the language in lecture 1. You don't *need* this lecture to write a record. You need it because weeks 5–10 will hand you a memory leak, a slow query, or a `StackOverflowError`, and "the JVM is a black box" is not a debugging plan. This lecture makes the box transparent enough to reason about.

## 1. From source to running code: the pipeline

Here's the whole journey of a Java program, end to end:

```
  Goal.java
     │  javac  (the compiler — ships in the JDK)
     ▼
  Goal.class   ── bytecode, a portable instruction set (NOT machine code)
     │  java  (the launcher — starts the JVM)
     ▼
 ┌──────────────────────────────────────────────────────┐
 │  JVM (HotSpot)                                        │
 │   1. classloader  — loads Goal.class into memory      │
 │   2. verifier     — checks the bytecode is well-formed │
 │   3. interpreter  — runs bytecode op by op (slow)      │
 │   4. JIT (C1, C2) — compiles HOT methods to native     │
 │   5. GC           — reclaims dead objects on the heap  │
 └──────────────────────────────────────────────────────┘
     │
     ▼
  native machine code for the hot paths + interpreted bytecode for the cold ones
```

Two surprises for people coming from C, Go, or Rust:

1. **`javac` does not produce machine code.** It produces *bytecode* — a compact, platform-independent instruction set defined by the JVM specification. A `.class` file built on your Mac runs unchanged on a Linux server, an ARM laptop, or a cloud container. "Write once, run anywhere" is exactly this: the bytecode is the "anywhere" artifact; the JVM is the per-platform piece.
2. **Compilation to native code happens at *run time*, not build time.** The JVM starts by *interpreting* bytecode (executing each instruction in software). As it runs, it notices which methods are "hot" (called a lot, or looping a lot) and hands those to the **JIT** — the Just-In-Time compiler — which produces optimized native machine code for your actual CPU. This is why Java programs "warm up": the first few thousand calls are interpreted; later calls run as native code.

## 2. Reading bytecode with `javap`

Bytecode isn't a mystery you have to take on faith. The JDK ships a disassembler, `javap`. Compile a tiny class and look.

```java
// Adder.java
public class Adder {
    public int add(int a, int b) {
        return a + b;
    }
}
```

```bash
javac Adder.java
javap -c Adder.class
```

You'll see roughly:

```
public int add(int, int);
  Code:
     0: iload_1        // push local variable 1 (a) onto the operand stack
     1: iload_2        // push local variable 2 (b) onto the operand stack
     2: iadd           // pop two ints, add them, push the result
     3: ireturn        // pop the int, return it
```

That's it. The JVM is a **stack machine**: instructions push and pop values from an operand stack rather than naming registers. `iload_1` pushes the int in local slot 1; `iadd` pops two and pushes their sum; `ireturn` returns it. Every method you write becomes a sequence like this.

You will not write bytecode by hand or memorize opcodes. But run `javap -c` on one of your own records this week. Seeing that `equals`, `hashCode`, and `toString` are *really there* in the `.class` file — generated by the compiler from your one-line `record` declaration — makes records click in a way no prose can.

```bash
# after mvn compile, disassemble a generated record
javap -c -p target/classes/com/crunch/tracker/domain/Goal.class
```

You'll find `equals`, `hashCode`, `toString`, and accessor methods you never wrote. The compiler emitted them. (In Java 21 they're backed by an `invokedynamic` to `java.lang.runtime.ObjectMethods`, so the bytecode is compact — but the behavior is exactly the hand-written `equals`/`hashCode` you'd otherwise maintain by hand.)

## 3. The interpreter and the JIT — why Java "warms up"

When the JVM first runs a method, it **interprets** the bytecode: a loop in the JVM reads each instruction and does what it says. Correct, portable, and relatively slow.

HotSpot watches counters. When a method has been called enough times (or a loop has spun enough times), it's deemed "hot" and queued for **JIT compilation**. HotSpot has two JIT compilers:

- **C1 (the client compiler):** compiles quickly, optimizes modestly. Gets you to native code fast.
- **C2 (the server compiler):** compiles slowly, optimizes aggressively — inlining, escape analysis, loop unrolling, dead-code elimination, devirtualization. Produces the fastest code.

This is **tiered compilation**: a method might be interpreted, then C1-compiled, then (if it stays hot) C2-compiled. The JVM can even *deoptimize* — throw away compiled code and fall back to the interpreter — if an assumption it bet on (say, "this method is never overridden") turns out to be wrong at run time.

The practical consequences you'll meet later:

- **Benchmarks must warm up.** Timing the first 100 iterations of a loop measures the interpreter, not your steady-state performance. Real benchmarks (you'll use JMH in a later week) run thousands of warm-up iterations first. If a teammate says "I timed it with `System.nanoTime()` around one call," their number is mostly JIT-warm-up noise.
- **Startup vs throughput is a real tradeoff.** A short-lived CLI never gets hot enough to benefit from C2; a long-running server does. This is why GraalVM Native Image and AOT exist (out of scope this week) — they trade peak throughput for instant startup by compiling ahead of time.

> **JVM Anatomy Quark #1.** Aleksey Shipilëv (a JVM engineer) writes a series of short essays each isolating one JVM behavior, with real measurements: <https://shipilev.net/jvm/anatomy-quarks/>. You don't need them this week. Bookmark them; by week 10 you'll be grateful they exist.

## 4. Memory: the heap and the stack

The JVM divides memory into regions. The two you must understand are the **heap** and the **stack**.

```
 ┌──────────────────────────────────────────────────────────┐
 │  JVM memory                                               │
 │                                                          │
 │   Heap (shared by all threads)                           │
 │   ┌────────────────────────────────────────────────┐    │
 │   │  every object: Strings, records, ArrayLists,     │    │
 │   │  arrays, your Goal/Habit/CheckIn instances       │    │
 │   │  — managed and reclaimed by the GC               │    │
 │   └────────────────────────────────────────────────┘    │
 │                                                          │
 │   Stacks (one per thread)                                │
 │   ┌──────────────┐  ┌──────────────┐                     │
 │   │ thread-1     │  │ thread-2     │   each frame holds:  │
 │   │  frame: main │  │  frame: run  │    - local variables │
 │   │  frame: foo  │  │  frame: ...  │    - the operand stack│
 │   │  frame: bar  │  │              │    - return address   │
 │   └──────────────┘  └──────────────┘                     │
 │                                                          │
 │   Metaspace: loaded class metadata (the .class info)     │
 └──────────────────────────────────────────────────────────┘
```

### The stack

Each thread has its own **stack**. Every time you call a method, the JVM pushes a **frame** holding that call's local variables, its operand stack (the `iload`/`iadd` machine from section 2), and where to return. When the method returns, its frame is popped. Stack memory is automatic and cheap — no GC involved.

**Primitives declared as locals** and **references (the pointers themselves)** live in the frame on the stack:

```java
void example() {
    int n = 5;                 // the int 5 lives in this frame on the stack
    Goal g = new Goal(1, ...); // the Goal OBJECT is on the heap;
                               // the reference 'g' (the pointer to it) is in the frame
}
```

When `example()` returns, `n` and the reference `g` vanish with the frame. The `Goal` object on the heap does *not* vanish immediately — it becomes eligible for garbage collection because nothing references it anymore.

**`StackOverflowError`** happens when you push too many frames — almost always unbounded recursion. Each call adds a frame; infinite recursion exhausts the (bounded) stack. The error names the runaway method in its trace.

### The heap

Every **object** — every `new` — lives on the **heap**, which is shared across all threads and managed by the garbage collector. Your `Goal`, your `ArrayList<Habit>`, every `String`, every record instance: heap. The variable holding it is just a reference (a pointer); the bytes of the object are on the heap.

```java
List<Goal> goals = new ArrayList<>();   // the ArrayList object: heap.
                                        //   'goals' (the reference): a local, on the stack.
goals.add(new Goal(1, "Read", 7));      // the Goal object: heap.
                                        //   the ArrayList now holds a reference to it.
```

**`OutOfMemoryError: Java heap space`** happens when the heap fills with objects the GC can't reclaim — because they're all still reachable (a leak: you kept adding to a `static List` and never cleared it) or because your working set genuinely exceeds `-Xmx`. You'll meet this in week 5 if a query loads a million rows into memory.

### Where does a record's data live?

A record is a reference type — a `Goal` instance is an object on the heap, like any other. Its `long id` and `int targetPerWeek` components are stored *inline inside that heap object*; its `String title` component is a reference to a separate `String` object, also on the heap. The reference you hold to the `Goal` is on the stack (if it's a local) or inside its containing object (if it's a field). Records don't change *where* things live — they change how much code *you* write to define the thing that lives there.

## 5. References, reachability, and `null`

A reference is a pointer to a heap object — or `null`, meaning "points at nothing."

```java
Goal g = new Goal(1, "Read", 7);  // g points at a heap object
g = null;                          // g now points at nothing; the object may be collected
```

The GC's core question is **reachability**: starting from a set of *GC roots* (local variables on every thread's stack, static fields, etc.), which heap objects can be reached by following references? Reachable objects are live. Unreachable objects are garbage — their memory can be reclaimed. You never call `free()`; you just stop referencing things, and the GC notices.

**`NullPointerException`** is the runtime error you get when you follow a `null` reference — call a method or read a field on `null`:

```java
String title = null;
int len = title.length();   // NullPointerException — you dereferenced null
```

Since Java 14, the JVM gives **helpful NPE messages** that name *exactly* which reference was null:

```
Exception in thread "main" java.lang.NullPointerException:
  Cannot invoke "String.length()" because "title" is null
```

That message — "because `title` is null" — tells you the culprit directly. Read it; don't guess. Lecture 1's `Optional` (and the exercises) exist precisely to keep `null` out of your APIs so this exception becomes rare. We'll wire `Optional` through the domain in the exercises and the mini-project.

## 6. Garbage collection — what it actually does

You never free memory in Java. The **garbage collector** finds unreachable objects and reclaims their space. Here's the model you need.

### Generational collection: the weak generational hypothesis

Decades of measurement support one observation: **most objects die young.** A `String` you build to format a log line, a temporary `List` inside a method, the intermediate objects in a loop — they become garbage almost immediately. A few objects (a cache, a long-lived service) live for the whole program.

GCs exploit this by splitting the heap into **generations**:

```
 Heap
 ┌─────────────────────────────┬───────────────────────────┐
 │  Young generation           │  Old generation           │
 │  ┌────────┬────────┬──────┐ │                           │
 │  │  Eden  │   S0   │  S1  │ │  long-lived objects that  │
 │  │ (new   │ (survivor      │  survived several young   │
 │  │ allocs)│  spaces)      │ │  collections              │
 │  └────────┴────────┴──────┘ │                           │
 └─────────────────────────────┴───────────────────────────┘
```

- **Eden:** where `new` objects are allocated. Fast — usually just bumping a pointer.
- **Survivor spaces (S0, S1):** objects that survive a young collection get copied here.
- **Old generation:** objects that survive several young collections get *promoted* (tenured) here.

A **minor GC** collects only the young generation. It's frequent and fast because most of Eden is dead — the collector copies the few survivors out and declares the rest of Eden free in one stroke. A **major / full GC** collects the old generation too; it's rarer and more expensive.

### G1 — the Java 21 default

The default collector in Java 21 is **G1 (Garbage-First)**. G1 divides the heap into many equal-size *regions* and labels each as Eden, Survivor, or Old. It collects the regions with the most garbage first (hence "garbage-first") and aims to keep pause times under a target you can set (`-XX:MaxGCPauseMillis`). It's a good general-purpose default — which is why it *is* the default. (Java also ships ZGC and Shenandoah for very-low-pause workloads, and the serial/parallel collectors for small or batch jobs. You pick a non-default collector only when you have a measured reason; this week, you don't.)

### Stop-the-world pauses

Most collectors must, for some phases, **stop the world** — pause all application threads — to safely move objects and update references. G1's pauses are short and bounded by design, but they're real. When someone says "the GC paused us for 200ms," they mean a stop-the-world phase ran long. Tuning GC is, largely, the art of keeping these pauses short and rare.

### Watching it happen

You can watch the GC work. Run any program with GC logging:

```bash
java -Xlog:gc -jar target/crunch-tracker.jar
```

You'll see lines like:

```
[0.123s][info][gc] GC(0) Pause Young (Normal) (G1 Evacuation Pause) 24M->5M(256M) 3.142ms
```

Read that: a young-generation pause moved the live set from 24M down to 5M, out of a 256M heap, in 3.1ms. That's the GC reclaiming ~19M of dead objects. Run your mini-project with this flag once this week (it's a stretch goal in the README) — seeing your own program's allocations get collected is the moment GC stops being abstract.

> **You do not tune GC this week.** You learn the model. The single practical takeaway for now: **allocating fewer short-lived objects in hot loops means less GC work.** You won't optimize for that yet — premature optimization is its own bug — but when week 5 shows you a slow endpoint allocating a million temporary objects, you'll know why it's slow and where to look.

## 7. Building and running pure Java with Maven

You won't run `javac` and `java` by hand for real projects — Maven does it. From week 1 you have a Maven skeleton. Here's the working set for a no-framework week.

```bash
# Compile sources to target/classes (this runs javac under the hood).
mvn compile

# Compile + run the unit tests (JUnit 5).
mvn test

# Run a main class (the exec plugin; configure mainClass in pom.xml).
mvn exec:java -Dexec.mainClass="com.crunch.tracker.App"

# Compile, test, and package into a runnable jar under target/.
mvn package

# Wipe target/ and rebuild from scratch.
mvn clean package
```

A minimal `pom.xml` for this week declares JDK 21 and pulls in JUnit 5:

```xml
<project xmlns="http://maven.apache.org/POM/4.0.0">
  <modelVersion>4.0.0</modelVersion>

  <groupId>com.crunch</groupId>
  <artifactId>crunch-tracker</artifactId>
  <version>0.1.0</version>

  <properties>
    <maven.compiler.release>21</maven.compiler.release>
    <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
  </properties>

  <dependencies>
    <dependency>
      <groupId>org.junit.jupiter</groupId>
      <artifactId>junit-jupiter</artifactId>
      <version>5.11.3</version>
      <scope>test</scope>
    </dependency>
  </dependencies>
</project>
```

`<maven.compiler.release>21</maven.compiler.release>` is the line that matters: it tells `javac` to target Java 21, so records, sealed types, and pattern-matching `switch` all compile. If you ever see "patterns in switch statements are a preview feature" or "records are not supported," your `release` is set below 21 — fix that line first.

We turn on lint this week so the compiler nags you toward modern idioms. Add to the compiler plugin config:

```xml
<compilerArgs>
  <arg>-Xlint:all</arg>
</compilerArgs>
```

Now `javac` warns on raw types, unchecked operations, and unnecessary casts — exactly the 2008-Java habits we're unlearning. Treat warnings as bugs.

## 8. A glance at what's *not* in this lecture

- **Threads and concurrency.** We mentioned per-thread stacks; the JVM's memory model, locks, and `java.util.concurrent` are a later topic. (Java 21's virtual threads are a headline feature, but they're out of scope this week.)
- **Classloading in depth.** We named the classloader; custom classloaders and module-path mechanics are advanced material.
- **GC tuning.** We described the model; choosing and tuning a collector is a production skill you'll grow into.
- **AOT / GraalVM Native Image.** We contrasted JIT with AOT in passing; native images are a week-10-and-beyond concern.

You have enough now to *reason* about all of these when they come up. That's the goal of a "what the JVM does" lecture: not to make you a JVM engineer, but to make the runtime legible.
